<objective>
Create the Phase 6 database foundation before any aggregation or anomaly logic lands: three Supabase tables, their indexes and RLS policies, plus the backend client changes required to write through service-role-only policies.

Purpose: Research shows the current backend mostly uses the anon Supabase client, while newer migrations rely on service-only RLS. If that mismatch is not fixed first, later Phase 6 plans can be implemented but will fail at runtime under real policies. This plan is also the Wave 0 validation foundation for the phase: later plans depend on the shared test helpers and service-client smoke coverage defined here.
Output: A human-runnable `database/migrations/data-pipeline.sql` migration, a backend Supabase client surface that can persist pipeline rows through RLS, and Wave 0 validation helpers/tests that subsequent plans can reuse.
</objective>
