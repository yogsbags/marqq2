<interfaces>
Recommended service surface from research:

```javascript
saveRawSnapshot(snapshotEnvelope)
aggregateSnapshot(snapshotRow)
upsertDailyKpis(rows)
```

Required KPI formulas:

```text
ctr  = clicks / impressions when impressions > 0 else NULL
cpl  = spend / leads when leads > 0 else NULL
cpa  = spend / conversions when conversions > 0 else NULL
cpc  = spend / clicks when clicks > 0 else NULL
roas = revenue / spend when spend > 0 else NULL
```
</interfaces>

Create `platform/content-engine/kpi-aggregator.js` as a small backend service in the existing `platform/content-engine/` style.

Required responsibilities:
- accept a raw snapshot envelope shaped for `connector_raw_snapshots`
- persist raw snapshots through the service-role-safe Supabase path from Plan 06-01
- extract supported metric atoms from payload JSON without normalizing every connector schema in the database layer
- compute canonical KPI rows for `company_kpi_daily`
- upsert rows on `(company_id, metric_date, source_scope)`
- expose a read helper for aggregated KPI rows that later API/anomaly code can reuse

Implementation constraints:
- do not introduce an ETL framework, ORM, or queue
- keep raw payload JSON opaque after metric extraction
- do not export any helper intended for agents to read raw tables
- prefer plain objects and small functions over class-heavy structure
- preserve SQL as the canonical storage surface even if extraction orchestration lives in JS
